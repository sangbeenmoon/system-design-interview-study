## 핵심 내용 요약

- SLA (Serive Level Agreement) : 서비스 사업자와 고객 사이에 맺어진 합의. SLA 에는 보통 서비스의 가용시간(uptime) 이 기술되어 있음.
- 많이 출제되는 개략적 규모 추정 문제는 QPS, 최대 QPS, 저장소 요구량, 캐시 요구량, 서버 수 등을 추정하는 것이다.

### 토큰 버킷 알고리즘
- 요청 하나당 토큰 한개 소모. 일정 시간마다 토큰 충전.
- 장점 : 구현 간단. burst of traffic 처리 가능.
- 단점 : 버킷 크기, 토큰 공급률 튜닝 까다로움.

### 누출 버킷 알고리즘
- 요청을 큐에 쌓아놓고, 처리율에 맞춰 요청 처리.
- 장점 : 큐의 크기가 제한되어 있어 메모리 사용량 측면에서 효율적. 안정적 처리가 필요할 때 적합.
- 단점 : burst of traffic 처리 불가. 

### 고정 윈도우 카운터 알고리즘
- 고정 윈도우 동안 요청 횟수를 카운트하고, 제한을 넘어간 요청은 해당 고정 윈도우 동안 block 처리.
- 장점 : 메모리 효율 좋음. 직관적.
- 단점 : 윈도우 경계 부근에서 요청이 몰리는 경우 제한보다 많은 양의 요청을 처리하게 됨.

### 이동 윈도우 로깅 알고리즘
- 요청이 들어오면 요청 시간을 로그로 남기고, 해당 요청 시간으로부터 윈도우를 그렸을 때 요청 횟수가 제한 보다 적으면 허용 , 많으면 block 처리.
- 장점: 고정 윈도우 카운터 알고리즘 단점 극복. 정교함.
- 단점: block 된 요청의 타임스탬프로 로그로 남겨야하기 때문에 비효율적인 메모리 사용.

### 이동 윈도우 카운터 알고리즘
- 현재 카운터 + (이전 카운터 요청 갯수 * 카운터 시간이 겹치는 비율) 로 현재 요청 갯수 계산
- 장점 : 메모리 효율 좋음. 고정 윈도우 카운터 알고리즘 단점 극복.
- 단점 : 직전 시간대 요청이 균등하게 분배되어 있다고 가정. (미미한 단점)

### 효과적 면접을 위한 4단계 접근법
- 1. 문제 이해 및 설계 범위 확정 : 요구사항과 가정을 분명히 하라.
  - 구체적으로 어떤 기능을 만들어야 하나?
  - 제품 사용자 수는 얼마나 되나?
  - 회사의 규모는 얼마나 빨리 커지리라 예상? 
  - 회사가 주로 사용하는 기술 스택? 설계 단순화를 위해 활용할 수 있는 기존 서비스로는 어떤 것이 있는가?
- 2. 개략적인 설계안 제시 및 동의 구하기 (high level design)
  - 핵심 컴포넌트를 포함하는 다이어그램을 그려라.
  - 필요하다면, 개략적인 추정 계산.
  - 필요하다면, api endpoint , db schema 설계.
  - 설계안에 대한 청사진을 제시하고 의견을 구하라.
- 3. 상세 설계 (deep dive)
  - 면접관과 설계 대상 컴포넌트 사이의 우선순위를 정하라.
  - algorithm
  - cache
  - DB
  - scaling
  - consistency
  - availability
- 4. 마무리
  - 면접관 후속 질문 or 스스로 추가 논의 진행.
  - bottleneck & trade-off
  - 오류 발생 시 무슨일이 생기는지.
  - 운영 이슈. (메트릭 수집, 모니터링, 로그, 배포)
  - 미래에 닥칠 규모 확장 대처 방안
